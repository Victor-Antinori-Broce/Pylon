## PASO 1: EL PUENTE ZOD ✅ COMPLETADO
## ═══════════════════════════════════════════════════════════════════════════

### Fecha: 2026-02-26

### Objetivo
Resolver el desfase de datos crítico entre Frontend (SvelteKit) y Backend (Hono)
estableciendo una única fuente de verdad compartida para la validación de Datasets.

### Problema Identificado
- Frontend esperaba: `DatasetDefinition` con `fields: FieldDefinition[]` (estructura anidada)
- Backend guardaba: `schema` como array plano sin validación estructural
- Resultado: Riesgo de corrupción de datos y contratos rotos entre capas

### Solución Implementada

#### 1. packages/shared/src/schemas.ts (NUEVO)
Creado el archivo central de esquemas Zod que define:

- `FieldTypeSchema` - Enum de 11 tipos de campo (text, number, boolean, date, etc.)
- `FieldDefinitionSchema` - Esquema completo de validación para cada campo:
  - Identificación: id, key, label, type
  - Validaciones: isRequired, isUnique, regex, min/max values/length
  - Relaciones: target, type (1:1/1:N/M:N), displayField
  - UI: width, hidden, order
- `DatasetDefinitionSchema` - Esquema maestro del dataset:
  - Campos base: id, name, slug, description, icon
  - **CRÍTICO**: `fields: FieldDefinitionSchema[]` - array validado de fields
  - Enterprise: policyJson, workflowJson
- `CreateDatasetPayloadSchema` / `UpdateDatasetPayloadSchema` - Para validación de endpoints
- Helpers: `validateDatasetDefinition()`, `safeValidateDataset()`, `createEmptyField()`

#### 2. Backend: apps/api/src/routes/datasets.ts (ACTUALIZADO)
Implementada validación estricta con @hono/zod-validator:

- POST `/datasets` - Valida contra `CreateDatasetPayloadSchema`
- PATCH `/datasets/:id` - Valida contra `UpdateDatasetPayloadSchema`
- Transformación de formatos: El nuevo formato (fields) se guarda en DB como:
  ```json
  {
    "fields": [...],
    "version": "2.0"
  }
  ```
- GET endpoints transforman la respuesta DB al formato DatasetDefinition estandarizado
- Rechazo automático con 400 Bad Request si el payload no cumple el esquema

#### 3. Frontend: apps/admin/src/types/datasets.ts (MIGRADO)
Re-exportación desde @gremius/shared:

- Todos los tipos ahora vienen de `import("@gremius/shared")`
- Se mantienen tipos UI-específicos: `FieldTypeOption`, `FIELD_TYPE_OPTIONS`
- Helpers legacy: `generateSlug()`, `generateFieldKey()`
- Notas de migración para desarrolladores futuros

### Dependencias Configuradas
- packages/shared: Agregado "zod": "^3.23.0"
- apps/api: Ya tenía @hono/zod-validator y zod
- apps/admin: Agregado "@gremius/shared": "workspace:*"

### Flujo de Datos Ahora
Frontend (SchemaBuilder.svelte)
  ↓ Crea payload con fields[]
POST /api/datasets
  ↓ @hono/zod-validator valida contra DatasetDefinitionSchema
  ↓ Si inválido: 400 Bad Request
  ↓ Si válido: Transforma a formato DB (schema: { fields, version: "2.0" })
DB PostgreSQL (JSONB)
  ↓
GET /api/datasets/:id
  ↓ Transforma DB → DatasetDefinition (extrae fields)
  ↓ Retorna JSON con estructura nested válida
Frontend recibe y renderiza

## PASO 2: EL ESTADO GLOBAL (SVELTE 5 RUNES) ✅ COMPLETADO
## ═══════════════════════════════════════════════════════════════════════════

### Fecha: 2026-02-26

### Objetivo
Implementar el manejo de estado global en el panel de administración usando
exclusivamente Svelte 5 Runes ($state, $derived), prohibiendo el uso de stores
antiguos (writable, readable, derived).

### Problema Identificado
- El panel de admin no tenía estado global centralizado
- Los componentes manejaban estado local disperso
- Se necesitaba un patr moderno sin boilerplate de stores

### Solución Implementada

#### 1. apps/admin/src/lib/states/datasets.svelte.ts (NUEVO)
Creado el gestor de estado global usando el patrón Clase Reactiva:

**Estado Reactivo con $state:**
- `datasets = $state<DatasetDefinition[]>([])` - Lista de datasets
- `isLoading = $state<boolean>(false)` - Estado de carga
- `selectedDataset = $state<DatasetDefinition | null>(null)` - Dataset activo
- `error = $state<string | null>(null)` - Error actual

**Estado Derivado con $derived:**
- `totalCount = $derived<number>(this.datasets.length)`
- `sortedDatasets = $derived<DatasetDefinition[]>([...sorted])`
- `hasDatasets = $derived<boolean>(this.datasets.length > 0)`

**Métodos CRUD:**
- `fetchDatasets()` - GET /api/datasets
- `fetchDatasetById(id)` - GET /api/datasets/:id
- `createDataset(payload)` - POST /api/datasets
- `updateDataset(id, payload)` - PATCH /api/datasets/:id
- `deleteDataset(id)` - DELETE /api/datasets/:id

**Instancia Singleton:**
```typescript
export const datasetStore = new DatasetStore();
```

### Características del Enfoque Runes

| Feature | Implementación | Beneficio |
|---------|---------------|-----------|
| Reactividad | `$state()` en propiedades de clase | Sin boilerplate de suscripciones |
| Computed | `$derived()` para valores calculados | Re-calculación automática |
| Global State | Instancia singleton exportada | Acceso directo desde cualquier componente |
| Type Safety | Tipos desde @gremius/shared | Autocompletado y validación |

### Uso en Componentes

```svelte
<script lang="ts">
  import { datasetStore } from '$lib/states/datasets.svelte';
  
  // Llama a la carga en mount
  $effect(() => {
    datasetStore.fetchDatasets();
  });
</script>

<!-- Reactividad fina: DOM se actualiza automáticamente -->
{#if datasetStore.isLoading}
  <LoadingSpinner />
{:else if datasetStore.error}
  <ErrorMessage message={datasetStore.error} />
{:else}
  {#each datasetStore.sortedDatasets as dataset}
    <DatasetCard {dataset} />
  {/each}
  
  <p>Total: {datasetStore.totalCount} datasets</p>
{/if}
```

```typescript
async function handleImageUpload(fieldKey: string, file: File) {
  uploadingStates[fieldKey] = true;
  
  try {
    const formDataPayload = new FormData();
    formDataPayload.append("file", file);
    
    const res = await fetch(`${API_URL}/api/media/upload`, {
      method: "POST",
      body: formDataPayload,
      credentials: "include"
    });
    
    const data = await res.json();
    if (!res.ok) throw new Error(data.error || "Upload failed");
    
    setField(fieldKey, data.url);
  } catch (err: any) {
    alert(err.message || "Failed to upload image");
  } finally {
    uploadingStates[fieldKey] = false;
  }
}
```

### Objetivo
Crear el SDK Universal (`@gremius/client`) que permita a cualquier desarrollador
hacer CRUD completo en 3 líneas de código, con tipado fuerte y compatible con
Edge, Browser y Node 18+.

### La Amenaza Competitiva
> Si un "Vibe Coder" o un LLM (como Claude o Cursor) no puede hacer un CRUD 
> completo en 3 líneas de código, abandonarán Gremius y se irán a Supabase.

### Solución Implementada

#### 1. packages/sdk/src/index.ts (SDK Principal)

**Clase GremiusClient:**
```typescript
export class GremiusClient {
  constructor(config: { url: string; apiKey?: string })
  
  collection<T>(name: string): CollectionRef<T>
}
```

**Clase CollectionRef (Patrón Fluent/Encadenamiento):**
```typescript
export class CollectionRef<T> {
  async get(options?: GetOptions): Promise<ListResponse<T>>
  async getById(id: string): Promise<T | null>
  async insert(data: Omit<T, 'id'>): Promise<T>
  async update(id: string, data: Partial<Omit<T, 'id'>>): Promise<T>
  async delete(id: string): Promise<void>
}
```

#### 2. Interfaz de Uso Final (La Promesa Cumplida)

```typescript
import { GremiusClient } from '@gremius/sdk';

const gremius = new GremiusClient({ 
  url: 'https://api.midominio.com', 
  apiKey: 'gremius_sk_123' 
});

// Leer
const posts = await gremius.collection('posts').get();

// Crear
const newPost = await gremius.collection('posts').insert({ title: 'Hola Mundo' });

// Actualizar y Borrar
await gremius.collection('posts').update(newPost.id, { title: 'Editado' });
await gremius.collection('posts').delete(newPost.id);
```

#### 3. Soporte TypeScript (Generics)

```typescript
interface User { 
  id: string; 
  name: string; 
  email: string;
}

// TypeScript infiere correctamente
const users = await gremius.collection<User>('users').get();
//    ^? ListResponse<User>

const user = await gremius.collection<User>('users').insert({ name: 'Alex' });
//    ^? User
```

## FASE DE RESCATE: Drizzle Where Conditions (App RLS) ✅ COMPLETADO
## ═══════════════════════════════════════════════════════════════════════════

### Fecha: 2026-02-26

### Misión
Ejecutar la **Fase de Rescate: Optimización de RAM y RLS de Aplicación**.

El motor de Gremius estaba cometiendo un pecado capital: traer TODOS los registros
de PostgreSQL a Node.js y aplicar `.filter()` en memoria basado en reglas JSON.
Esto agotaría la RAM del servidor con unos pocos miles de registros.

### El Problema (Antipatrón Mata-RAM)

```typescript
// ANTES (Ineficiente - Peligroso)
const allEntries = await db.select().from(dataEntries); // 100,000 filas
const filtered = allEntries.filter(entry => {            // En memoria
  return entry.data.authorId === user.id;
});
// Resultado: Node.js OOM (Out of Memory) + Alta latencia
```

### La Solución (Drizzle Application RLS)

```typescript
// DESPUÉS (Optimizado - Seguro)
const whereCondition = buildDrizzleWhere(rules, user);
const filtered = await db
  .select()
  .from(dataEntries)
  .where(whereCondition); // PostgreSQL filtra, no Node.js
// Resultado: Solo 5 filas viajan por la red
```

### Archivos Creados/Modificados

#### 1. `apps/api/src/lib/rls-builder.ts` (NUEVO)

Módulo central del traductor de permisos:

```typescript
// Funciones principales
export function buildDrizzleWhere(
  rules: RLSRule[] | undefined,
  user: RLSUser
): SQL | undefined

export function buildDatasetWhere(
  datasetId: string,
  rules: RLSRule[] | undefined,
  user: RLSUser,
  dataColumn?: string
): { where: SQL | undefined; hasRLS: boolean }

export function checkReadAccess(policy: PolicyJson | null, user: RLSUser): boolean
export function checkWriteAccess(policy: PolicyJson | null, user: RLSUser): boolean
```

**Operadores soportados:**
- `eq` → `field = value`
- `neq` → `field <> value`
- `contains` → `field LIKE '%value%'`
- `gt` → `field::numeric > value`
- `lt` → `field::numeric < value`

**Variable mágica:** `__USER_ID__` se reemplaza por `user.id`

#### 2. `apps/api/src/routes/datasets.ts` (REFACTORIZADO)

Endpoint GET `/:id/entries` ahora usa RLS eficiente:

```typescript
dataSetsFullRoutes.get("/:id/entries", async (c) => {
  // 1. Obtener política del dataset
  const [dataset] = await db
    .select({ policyJson: dataSets.policyJson })
    .from(dataSets)
    .where(eq(dataSets.id, datasetId))
    .limit(1);
  
  // 2. Verificar acceso
  if (!checkReadAccess(policy, user)) {
    return c.json({ error: "Unauthorized" }, 403);
  }
  
  // 3. Construir WHERE con RLS (¡La magia!)
  const { where, hasRLS } = buildDatasetWhere(
    datasetId,
    policy?.rlsRules,
    user
  );
  
  // 4. Query optimizada - PostgreSQL hace el trabajo
  const docs = await db
    .select()
    .from(dataEntries)
    .where(where!)
    .orderBy(dataEntries.sortOrder)
    .limit(limit);
  
  return c.json({ 
    docs, 
    totalDocs: docs.length,
    _meta: { rlsEnabled: hasRLS, filteredBy: "database" }
  });
});
```

#### 3. `apps/api/src/middleware/permissions.ts` (ACTUALIZADO)

- Delega lógica a `rls-builder.ts` para consistencia
- Marca `createRLSFilter()` como **deprecated**
- Mantiene compatibilidad con código legacy

### Beneficios

| Aspecto | Antes | Después |
|---------|-------|---------|
| **Memoria** | O(N) - Todas las filas | O(resultados) - Solo filtradas |
| **Latencia** | Alta (transferencia) | Baja (filtrado en DB) |
| **Escalabilidad** | Colapsa con 100k+ filas | Funciona con millones |
| **Seguridad** | RLS en memoria (bypassable) | RLS en SQL (nativo) |
| **CPU** | Filtrado en Node.js | Filtrado en PostgreSQL |

### Criterios de Aceptación Verificados

✅ **Feature: Traductor de Permisos Dinámicos**
```gherkin

  Then buildDrizzleWhere reemplaza __USER_ID__ por 'user-123'
  And devuelve condición SQL: data->>'author_id' = 'user-123'
```

✅ **Feature: Eliminación de Filtrado en Memoria**

```gherkin
  And Node.js solo recibe 5 registros en memoria
```

### Notas Técnicas

1. **Acceso a campos JSONB:** Usamos `data->>'fieldName'` para acceder a campos
dentro de la columna JSONB "data" en PostgreSQL.

2. **Operadores numéricos:** Para `gt` y `lt`, hacemos casting a numeric:
`(data->>'field')::numeric > value`

3. **Combinación de reglas:** Todas las reglas se combinan con `AND`:
`AND(rule1, rule2, rule3)`

4. **Backwards compatibility:** `createRLSFilter()` sigue disponible pero marcado
como deprecated para no romper código existente.

## FASE DE PULIDO UI Y SETTINGS ✅ IMPLEMENTADO
## ═══════════════════════════════════════════════════════════════════════════

### Fecha: 2026-03-01

### Objetivo
Corregir errores visuales de consistencia, asegurar la navegación inicial,
estandarizar la tipografía en menús y crear el andamiaje visual 
para la nueva vista de Configuración usando Svelte 5.

### Mejoras Ejecutadas

#### 1. Consistencia del Tema Oscuro (New Formula Modal)
- **Problema:** Los inputs en `FormulasView.svelte` usaban clases de Tailwind que podían desentonar o resultar en fondos brillantes que rompían el modo oscuro por contraste.
- **Solución:** Se aplicaron clases explícitas `bg-gray-800 text-white placeholder-gray-500 border-gray-700` a los campos para garantizar la armonía visual.

#### 2. Enrutamiento y Textos (Rutas y Sidebar)
- **Problema 1:** La ruta `GET /datasets/new` estaba vacía/rota.
- **Solución 1:** Se recreó la vista `apps/admin/src/routes/datasets/new/+page.svelte` como una página dedicada con botón de retorno, consumiendo y renderizando internamente al componente `<SchemaBuilder />`. La respuesta y los flujos asíncronos enlazan directamente con `datasetStore.createDataset()`.
- **Problema 2:** Errores de capitalización de la tipografía ("kpi", "workers") en el UI.
- **Solución 2:** Modificadas en `SidebarLayout.svelte`, estandarizando los labels a "KPIs" y "Workers" explícitamente y mejorando su peso.

#### 3. Formulario Placeholder en Settings
- **Problema:** La pestaña de Configuración General en `Settings.svelte` mostraba el mensaje provisional *"Site configuration coming soon"*.
- **Solución:** Se construyó un formulario estructurado con inputs reales (Site Name, Logo URL, Default Metadata) soportados por los Svelte 5 Runes (`$state`). Se instaló la simulación visual sin escribir el backend de guardado, limitando el alcance a requerimientos de maquetación pura.

### Criterios de Aceptación Cumplidos
✅ **Scenario: Consistencia del Tema Oscuro en Modales** (Inputs del modal New Formula usan colores grises oscuros explícitos integrados al tema)
✅ **Scenario: Acceso a nuevo Dataset** (Ruta `/datasets/new` expone y compila exitosamente el editor de esquemas sin 404)
✅ **Scenario: Capitalización en la UI** (Menú lateral capitaliza explícitamente KPIs y Workers)
✅ **Scenario: Renderizado de campos de configuración** (Formulario y estados visuales "loading" construidos, sin placeholder "coming soon")

## Diagnóstico Inicial

### Estado encontrado:
Las 5 piezas del Core ya tenían carpetas en packages/core/:
  1. packages/core/data-sets/         ← Data Sets & Visualizer
  2. packages/core/media-library/     ← Media Library
  3. packages/core/page-builder/      ← Page Builder
  4. packages/core/promote-to-content/ ← Promote to Content
  5. packages/core/webhooks/          ← Webhooks Engine

### Problemas detectados:

#### P1: Archivos muertos en packages/core/ (CRÍTICO)
Las carpetas media-library, data-sets, webhooks y promote-to-content
contenían copias IDÉNTICAS de archivos en apps/api/src/ con imports
relativos rotos (../../db apuntaba a packages/core/db que NO EXISTE).
Solo page-builder funcionaba correctamente (es autocontenido).

#### P2: Module-loader no resolvía desde packages/core/
Para webhooks y promote, el loader intentaba packages/grimoires/
(no existe), generaba error de DEBUG, y caía a apps/api/src/modules/.
Funcionaba, pero por la ruta incorrecta.

#### P3: Feature Flags desconectados del sidebar
gremius.config.ts controlaba montaje de rutas en Hono, pero el
endpoint /api/modules (que alimenta el sidebar) usaba estado de DB
sin consultar los feature flags. Si webhooks.enabled=false en config,
el sidebar seguía mostrando Webhooks como habilitado.

#### P4: Entradas faltantes en sidebar
Page Builder y Promote to Content no aparecían en la navegación
del panel admin de SvelteKit.

## Correcciones Aplicadas

### Fix 1: Limpieza de packages/core/ (archivos muertos)
- Eliminados todos los .ts duplicados con imports rotos
- Actualizados index.ts como re-exports canónicos:
  * packages/core/media-library/index.ts → re-exporta desde apps/api/src/routes/media.ts
  * packages/core/data-sets/index.ts → re-exporta desde apps/api/src/routes/datasets.ts
  * packages/core/webhooks/index.ts → re-exporta desde apps/api/src/modules/webhooks/
  * packages/core/promote-to-content/index.ts → re-exporta desde apps/api/src/modules/promote/
  * packages/core/page-builder/index.ts → autocontenido (sin cambios)

### Fix 2: apps/api/src/index.ts importa a través de packages/core/
- Media Library y Data Sets ahora se importan vía packages/core/:
  import { mediaRoutes, uploadRoutes } from "../../../packages/core/media-library";
  import { dataSetsFullRoutes, explorerRoutes } from "../../../packages/core/data-sets";

### Fix 3: Module-loader resuelve Core antes que Grimoires
- Agregado CORE_DIR_MAP con mapeo: webhooks → webhooks, promote → promote-to-content
- El loader ahora intenta packages/core/{dir} PRIMERO para módulos core
- Fallback a packages/grimoires/ y apps/api/src/modules/ se mantiene

### Fix 4: Feature Flags sincronizados con /api/modules
- Importado gremiusConfig en routes/modules.ts
- Creado CORE_FLAG_MAP: media-library, promote, webhooks, blocks → flags del config
- transformModule() ahora usa configFlag como override prioritario sobre DB/defaults
- Resultado: si gremius.config.ts dice webhooks.enabled=false, /api/modules
  responde enabled:false y el sidebar oculta la pestaña Webhooks

### Fix 5: Sidebar completado con las 5 piezas
- Agregados iconos Blocks y Megaphone de lucide-svelte
- Nuevas entradas en allNavItems:
  * /builder → Page Builder (moduleKey: "blocks")
  * /promote → Promote to Content (moduleKey: "promote")
- Creada página placeholder: apps/admin/src/routes/promote/+page.svelte
